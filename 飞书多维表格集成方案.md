# AI 智能客服系统 — 完整技术方案

> 基于飞书多维表格 + AI 对话的智能客服平台，支持用户端对话与管理端运营。

---

## 一、项目概述

### 1.1 产品定位

为学员提供 AI 智能问答服务，管理员可进行数据分析和系统运营管理。

### 1.2 核心功能

| 子系统 | 功能 |
|--------|------|
| **用户端** | 注册/登录、AI 对话（流式输出）、3次免费→付费订阅、对话记录双写（飞书+MySQL） |
| **管理端** | 数据分析仪表盘、飞书/企微接口配置、AI 大模型 API 配置、Token 计费监控、用户订阅管理、兑换码管理、公告设置、禁用词设置、到期提醒、支付设置、注册设置 |

### 1.3 规模要求

- 预计 **2000 人同时在线**
- AI 对话请求峰值约 200-500 并发

## 二、系统架构

### 2.1 技术栈

| 层 | 技术选型 | 说明 |
|----|----------|------|
| 前端框架 | **Vue3 + Vite + Pinia** | 用户端 + 管理端两个 SPA |
| 用户端 UI | TailwindCSS | 聊天界面，简洁现代 |
| 管理端 UI | Element Plus + ECharts | 后台管理，表格/图表丰富 |
| 后端 | **FastAPI + uvicorn** | 原生 async，支持 SSE 流式输出，并发性能远超 Flask |
| ORM | SQLAlchemy 2.0 (async) | 异步数据库操作 |
| 数据库 | **MySQL 8.0** | 主数据存储 |
| 缓存/队列 | **Redis** | 会话管理、计数器、限流、飞书 token 缓存、异步任务队列 |
| AI | Claude API (streaming) | 流式对话 |
| 飞书 | 飞书开放平台 API | 多维表格读写 |

### 2.2 架构图

```
┌──────────────────┐     ┌──────────────────┐
│   Vue3 用户端     │     │   Vue3 管理端     │
│  (chat + login)  │     │  (dashboard)     │
└────────┬─────────┘     └────────┬─────────┘
         │                        │
         └───────────┬────────────┘
                     │ HTTP / SSE
             ┌───────▼────────┐
             │  FastAPI 后端   │
             │  (uvicorn)     │
             └──┬───┬───┬──┬─┘
                │   │   │  │
         ┌──────┘   │   │  └──────┐
         ▼          ▼   ▼         ▼
      MySQL      Redis  Claude   飞书 API
     (主存储)  (缓存)  (AI)   (异步同步)
```

### 2.3 为什么选 FastAPI 而非 Flask

| 对比项 | Flask | FastAPI |
|--------|-------|---------|
| 并发模型 | 同步阻塞（需 gevent 补丁） | 原生 async/await |
| 2000并发 | 需大量 worker，内存暴涨 | uvicorn 轻松处理 |
| SSE 流式输出 | 需手动实现 | 原生 StreamingResponse |
| API 文档 | 需手动写或用 flask-restx | 自动生成 OpenAPI 文档 |
| 学习成本 | 低 | 低（语法与 Flask 相似） |

## 四、数据库设计（MySQL）

### 4.1 用户表 `users`

```sql
CREATE TABLE users (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    phone           VARCHAR(20) UNIQUE,
    email           VARCHAR(100) UNIQUE,
    password_hash   VARCHAR(255) NOT NULL,
    nickname        VARCHAR(50),
    avatar_url      VARCHAR(500),
```

### 5.2 飞书附件处理

```json
{
  "方案截图": [
    {
      "file_token": "...",
      "name": "screenshot.png",
      "size": 123456,
      "tmp_url": "https://...",
      "type": "image/png",
      "url": "https://..."
    }
  ]
}
```

**图片访问方案**：后端代理模式

- 前端请求 `/api/image/{file_token}`
- 后端用 `tenant_access_token` 实时拉取图片，pipe 返回给前端
- 优点：URL 永不过期，不暴露 token，支持聊天记录回看
- `tenant_access_token` 缓存在 Redis（2小时 TTL，提前5分钟刷新）

> 不使用 `tmp_url` 方案，因为 2小时过期后用户无法回看历史消息中的图片。

### 5.3 飞书数据双写策略

```
用户发消息/AI回复
    │
    ├── 同步写 MySQL（主存储，必须成功）
    │
    └── 异步写飞书（通过 Redis 队列，允许延迟/重试）
         │
         ├── 根据 feishu_routes 配置匹配目标表格
         ├── 调用飞书 API 写入
         ├── 成功 → 更新 messages.feishu_synced = 1
         └── 失败 → 重试3次，仍失败则记录错误日志
```

**飞书 API 限流应对**：
- 飞书默认 100 QPS，高峰期可能不够
- 异步队列自带削峰能力，可配置写入速率
- 失败重试间隔：1s → 5s → 30s

## 六、核心功能详细设计

### 6.1 用户注册/登录

| 项 | 说明 |
|----|------|
| 注册方式 | 手机号/邮箱 + 密码 |
| 密码存储 | bcrypt 哈希 |
| 登录凭证 | JWT (存 Redis，支持主动注销) |
| Token 有效期 | access_token 2h，refresh_token 7d |

### 6.2 AI 对话流程

```
用户发送消息
    │
    ├── 1. 鉴权：校验 JWT
    ├── 2. 配额检查（Redis 原子操作）：
    │     - free 用户：free_chats_left > 0？
    │     - 订阅用户：subscribe_expire > now？
    ├── 3. 扣减配额（Redis DECR，异步同步到 MySQL）
    ├── 4. 加载 FAQ 知识库（Redis 缓存，10分钟 TTL）
    ├── 5. 加载文档索引（Redis 缓存，10分钟 TTL）
    │     - 文档数量 <50 条：全量传给 Claude
    │     - 文档数量 >50 条：先按关键词预筛选 top 10
    ├── 6. 构建 Claude 请求：
    │     - system_prompt（从 ai_config 表读取）
    │     - FAQ 上下文
    │     - 候选文档列表
    │     - 用户消息（含图片则用 Vision）
    ├── 7. 调用 Claude API（streaming）
    │     - 通过 SSE 流式返回给前端
    ├── 8. 流结束后：
    │     - 解析完整回复 + usage（input/output tokens）
    │     - 同步写 MySQL（messages + token_usage）
    │     - 异步写飞书（Redis 队列）
    └── 9. 返回结构化响应
```

### 6.3 AI 回复结构化格式

统一为 JSON，前端按类型分别渲染：

```json
{
  "text": "回复正文内容...",
  "images": [
    { "url": "/api/image/xxx", "name": "方案截图1.png" }
  ],
  "docs": [
    { "title": "操作指南", "link": "https://...", "tip": "请参考第3节" }
  ]
}
```

### 6.4 学员图片与 FAQ 图片比对（P1）

- 学员上传图片 → 文本初步匹配 2-3 条 FAQ → 取出 FAQ 的"问题截图"
- 将学员图 + FAQ 示例图一起传给 Claude Vision（最多 3 张 FAQ 图，控制 token 消耗）
- 仅当文本匹配得分较高时才触发，避免无效调用

## 十二、MVP 开发清单

### 12.1 首版上线目标

首版目标不是一次性完成全部运营能力，而是先完成可用闭环：

- 用户可以注册/登录
- 用户可以发起 AI 对话并看到流式回复
- 每个新用户有 3 次免费对话次数
- 免费次数用完后，管理员可通过后台人工开通订阅或发放兑换码
- 所有问答记录同步写入 MySQL，并异步写入飞书
- 管理员可以查看基础数据、调整 AI 配置、查看 Token 用量

### 12.2 首版必须实现范围（MVP）

| 模块 | 是否首版必须 | 说明 |
|------|--------------|------|
| 用户注册/登录 | 是 | 支持手机号或邮箱登录，JWT 鉴权 |
| 注册设置 | 是 | 管理员可控制注册开关、注册方式、是否需要邀请码、默认免费次数 |
| AI 对话 | 是 | 支持文本问答，SSE 流式输出 |
| 免费次数控制 | 是 | 每个新用户默认 N 次（由注册设置决定），需防并发重复扣减 |
| FAQ 知识库 | 是 | 从飞书读取 FAQ，命中后可返回方案截图 |
| 文档推荐 | 是 | 从飞书文档索引表筛选候选文档，返回结构化链接 |
| 禁用词过滤 | 是 | 用户发送消息时过滤敏感词，管理员可维护词库 |
| MySQL 主存储 | 是 | 用户、会话、消息、Token 消耗、订阅状态 |
| 飞书异步同步 | 是 | 对话记录异步写入飞书，不阻塞用户请求 |
| 管理员登录 | 是 | 后台基础鉴权 |
| 数据分析总览 | 是 | 用户数、对话数、热门问题、Token 总量 |
| 大模型 API 设置 | 是 | 多模型管理、默认模型切换、提示词、温度、开关项 |
| 公告设置 | 是 | 发布/编辑/下架公告，用户端展示 |
| 兑换码管理 | 是 | 支持批量生成与核销 |
| 订阅管理 | 是 | 管理员手动开通/延期 |

### 12.3 首版建议暂缓功能

| 模块 | 暂缓原因 |
|------|----------|
| 自动支付接入（微信/支付宝） | 接口联调和回调验签复杂，建议放到第二阶段 |
| 学员图片与 FAQ 图片智能比对 | Vision 成本高，流程复杂，先保证文本问答闭环 |
| 多飞书租户复杂路由 UI | 首版可先支持单套默认飞书配置，后台保留扩展字段 |
| 细粒度管理员权限 RBAC | 首版先用 `super` / `normal` 两级角色 |
| 高级 BI 分析 | 首版先做基础总览和趋势统计 |
| 到期提醒设置 | 多渠道提醒需对接短信/邮件服务，首版先做站内提示，后续补充 |
| 企业微信接口设置 | Webhook 推送属于运营增强，非核心流程，可后续迭代 |

### 12.4 首版验收标准

| 项目 | 验收标准 |
|------|----------|
| 登录注册 | 用户可以完成注册、登录、退出、刷新 Token |
| AI 对话 | 连续 20 次测试对话无异常，支持流式输出 |

## 十三、API 与数据字典

### 13.1 通用响应格式

除 SSE 流式接口外，所有 HTTP 接口统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "2d7f7c2a-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
}
```

字段约定：

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | number | `0` 表示成功，非 `0` 表示失败 |
| `message` | string | 提示信息 |
| `data` | object | 业务数据 |
| `request_id` | string | 请求追踪 ID，便于日志排查 |

### 13.2 用户注册接口 `POST /api/auth/register`

请求体：

```json
{
  "phone": "13800138000",
  "email": "user@example.com",
  "password": "12345678",
  "nickname": "张三"
}
```

请求字段说明：

| 字段 | 必填 | 类型 | 说明 |
|------|------|------|------|
| `phone` | 否 | string | 手机号，和 `email` 至少传一个 |
| `email` | 否 | string | 邮箱，和 `phone` 至少传一个 |
| `password` | 是 | string | 明文密码，后端 bcrypt 加密 |
| `nickname` | 否 | string | 昵称 |

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "user_id": 1,
    "free_chats_left": 3
  },
  "request_id": "req_xxx"
}
```

#### FAQ 知识库表映射

#### 文档索引表映射

|----------|------|--------------|------|

| `飞书文档内容` | Text | `doc.summary` | 文档召回与提示上下文 |
| `温馨提示` | Text | `doc.tip` | 回复补充说明 |

#### 对话记录表（建议新增）

为满足“所有问题和 AI 回答都写入飞书”的要求，建议新增一张飞书多维表格：`对话记录表`。

| 飞书字段 | 类型 | MySQL 来源字段 | 说明 |
|----------|------|----------------|------|
| `用户ID` | Text | `users.id` | 用户唯一标识 |
| `会话ID` | Text | `conversations.id` | 会话标识 |
| `用户消息ID` | Text | `messages.id`（role=`user`） | 用户消息 ID |
| `AI消息ID` | Text | `messages.id`（role=`assistant`） | AI 消息 ID |
| `用户问题` | Text | `messages.content`（user） | 用户原问题 |
| `AI回复` | Text | `messages.content`（assistant） | AI 完整回复 |
| `推荐文档` | Text | `messages.docs` | 建议转为 JSON 字符串或换行文本 |
| `方案截图链接` | Text | `messages.images` | 首版建议只存 URL 文本，不回传附件 |
| `模型` | Text | `token_usage.model` | 使用的 AI 模型 |
| `输入Token` | Number | `token_usage.input_tokens` | 输入消耗 |
| `输出Token` | Number | `token_usage.output_tokens` | 输出消耗 |
| `费用USD` | Number | `token_usage.cost_usd` | 本次请求成本 |
| `创建时间` | DateTime | `messages.created_at` | 记录时间 |

> 首版建议：飞书中只做“记录留痕”和运营查看，不把结构复杂的 `docs` / `images` 反向转为飞书附件，优先保证写入稳定性。

## 十四、错误处理与降级策略

| 场景 | 策略 |
|------|------|
| Claude API 超时/失败 | 返回友好提示"AI 暂时繁忙，请稍后重试"，不扣费 |
| 飞书 API 限流 | 异步队列自动延迟重试，不影响用户体验 |
| 飞书 token 过期 | Redis 缓存提前 5 分钟刷新，失败时实时重新获取 |
| MySQL 写入失败 | 返回错误，不继续写飞书 |
| Redis 不可用 | 降级为 MySQL 直查，性能下降但功能不中断 |
| 用户并发超限 | 接口限流，返回 429 |
